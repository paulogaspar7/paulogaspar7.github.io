The /modernize‑service
assembly line.

One command takes a legacy Spring Boot service and drives it through a complete modernization — internal‑libs 4.x migration, observability, business metrics, REST surfaces over Thrift/gRPC, OpenAPI docs, tests, and Grafana — by orchestrating a fleet of specialized Claude Code agents. Every step runs an execute‑validate cycle until it passes or hits its retry ceiling.

Claude Opus 4.5 Claude Code early‑2026 build requires common + jmod plugins 13 operations · 26 agents
▲

Historical context. This workflow was authored for a Claude Code build from late 2025 and was run with Claude Opus 4.5 as the orchestrator. Both marketplaces in this repository must be installed: common (shared documentation & validation tooling) and jmod (the Java Services Modernization pipeline, which depends on Common). The batch variant additionally needs the Agent Teams feature, which was experimental on earlier Claude Code versions.

02

The engine: every operation is a cycle

Each box in the pipeline above is not a single shot — it is an execute‑validate loop. An ag-* executor does the work and writes an implementation report; a dedicated agchk-* validator independently checks it against a criteria skill. On failure the executor reads the findings and retries, up to the operation's max cycles. Agents talk to each other through files in ai-lastbuild-reports/ — the orchestrator only reads verdicts.

protocol · execute → validate → retry
executor

ag-*

Opus-class implementation agent. Preloads its howto-* + criteria-* skills, makes the changes, writes {op}-implementation.md.

validator

agchk-*

Independent checker. Preloads the same criteria-* skill plus a Python deterministic validator, then returns PASS/FAIL.

channel

ai-lastbuild-reports/

File-based protocol. Implementation & validation reports persist here so retries carry context and the orchestrator stays lean.

04

Phase by phase

All operations run in verify-and-enhance mode — nothing is skipped just because it "already exists", because the underlying skills may have improved since the last run.

Phase 0

Research

Three read-only detectors (project-analyzer, thrift-detector, grpc-detector) plus a read of the project's CLAUDE.md run in parallel. Findings are summarised back to the user.

Group A · critical

Foundation

Sequential. The internal-libs 4.x migration must PASS before anything else; then the shade module and REST-over-protocol layers are added. A hard failure here stops the run.

Group B · parallel

Capabilities

Five operations launch at once once op1 is green: REST client, DB-test migration, observability, business metrics, and OpenAPI — everything that depends only on the migration.

Group C · parallel

Surface & visibility

REST tests, REST API docs, and the local Grafana stack fan out after Group B. The standardized dashboard (op11) waits for both metrics (6a) and the Grafana stack (10).

Group D · docs

Documentation

common:docproj-assessor grades documentation maturity, then the matching docproj-* writer agents create/update README, CLAUDE.md and docs/*. They self-validate.

Final gate

Build & summary

A full mvn clean then mvn clean verify from the project root, a per-operation status table, and the build verdict — saved to modernize-service-summary.md.

05

What happens when something fails

ScenarioActionWhy
Operation 1 fails after max cyclesSTOP — ask the user how to proceedThe internal-libs migration is the foundation; the run cannot continue safely without it.
Operations 2–11 fail after max cyclesLOG WARNING with validator findings, continueNon-blocking gaps — better to finish with known issues than abort the whole modernization.
Condition not metMark SKIPPEDNormal flow — not every feature applies to every service.
Validator agent errorTreat as FAIL for that iterationIf validation can't run, quality is not assured.
Executor agent errorTreat as FAIL for that iterationIf implementation didn't complete, the work isn't done.
Single critical path. Only Operation 1 can halt the pipeline. Everything downstream degrades gracefully into warnings, so the orchestrator always reaches the final build gate and produces a summary.

07

Scaling out: /modernize‑all‑services

The same pipeline, run across a whole fleet of services in parallel using Agent Teams. A team lead creates one task per service, spawns up to three background ag-modernize-worker agents, and each worker claims tasks from a shared list. Per service, a worker runs the full modernize pipeline, then an improvement cycle, then writes a combined report.

teams · /jmod:modernize-all-services ubag ucontent …

Inside Phase 2 — the improvement cycle

Where the per-operation agchk-* validators check one operation each, the improvement cycle verifies interactions between operations: that the core business logic was not changed, that every Thrift/gRPC method has a REST endpoint, and that every REST endpoint has a client method.

howto-improve-service · max 3 iterations
  • Workers self-coordinate through the shared task list — they claim the lowest pending task, set themselves as owner, and message the lead with a verdict when done.
  • Worker subagents are not team members — each worker spawns the ag-* pipeline agents as ordinary subagents (no team_name).
  • Graceful shutdown — on a shutdown request a busy worker finishes its current service first, then the lead runs TeamDelete() and assembles the fleet-wide report.
